Implementation plan / secure preview hosting

Agent HTML Plan Host

A small service where an agent can upload a single HTML plan, receive a short-lived URL, and show it to a user without giving the uploaded document access to credentials, cookies, internal networks, or other tenants.

Recommended Shape

Keep the service intentionally boring: accept HTML, validate it, store it as immutable content, serve it from a separate preview origin, then delete it automatically.

Agent API

Authenticated agents upload HTML through a narrow API. The API returns the preview URL and stores metadata: owner, creation time, expiry, size, hash, and policy version.

UPLOAD →

Object Store + Metadata

HTML is written once under a random ID. Metadata lives in D1, SQLite, Postgres, or KV depending on deployment. Old plans expire through a scheduled worker.

SERVE →

Preview Origin

Plans render at a cookieless domain such as plans.example.net with strict response headers and no access to the upload/admin API origin.

API Contract

Optimize for agents: one request in, one URL out. Add optional controls without making the happy path noisy.

1Upload

POST /v1/plans accepts text/html or JSON with html, title, and optional ttl_seconds.

2Return

Response includes id, url, expires_at, and sha256. Agents only need to show the URL.

3Manage

Optional DELETE /v1/plans/:id and GET /v1/plans/:id for revocation, audit, and UI integrations.

Security Model

The risky part is not storage; it is rendering untrusted HTML. Treat every uploaded plan as hostile content even when it came from one of your own agents.

Hard Requirements

  • Serve previews from a separate domain with no auth cookies.
  • Require bearer tokens or mTLS-equivalent trust for upload endpoints.
  • Enforce max size, max TTL, per-agent rate limits, and content type checks.
  • Use random, non-enumerable IDs with at least 128 bits of entropy.
  • Set Content-Security-Policy to block scripts by default.
  • Set Cross-Origin-Opener-Policy, X-Content-Type-Options, and Referrer-Policy.
Risk Control
XSS against your app Never serve uploaded HTML from the same origin as the API or dashboard.
Data exfiltration Default CSP: default-src 'none'; img-src data: https:; style-src 'unsafe-inline'; sandbox.
Phishing or stale links Visible banner with owner, creation time, expiry, and revoke button in admin UI.
Storage abuse Short default TTL, quota by agent, object size cap, and scheduled deletion.
Homelab exposure Put public traffic behind Cloudflare Tunnel or reverse proxy with isolated container network.

Deployment Options

Both Cloudflare and a homelab can work. I would start with Cloudflare unless you need local filesystem control, private-only previews, or zero external dependency.

Recommended MVP

Cloudflare

  • Worker handles upload API and preview serving.
  • R2 stores HTML objects; D1 or KV stores metadata.
  • Cloudflare Access can protect admin routes.
  • Scheduled Worker deletes expired content.
  • Fast public URLs without opening a home network port.
Good for private infra

Homelab

  • Small Go, Rust, Node, or Python service in Docker.
  • SQLite metadata plus local disk or MinIO object storage.
  • Caddy or Traefik terminates TLS and adds security headers.
  • Cloudflare Tunnel can expose only the preview/API ports.
  • More operational responsibility for backups and patching.

MVP Build Plan

Build the thin vertical path first, then tighten policy and management around it.

Upload and Serve

Implement POST /v1/plans, generate IDs, store HTML, and return public URLs. Add a small CLI example agents can call with curl.

Isolation and Headers

Split API and preview origins. Add strict CSP, sandboxing, no-sniff, referrer policy, and cache headers. Confirm cookies are never sent to preview pages.

Auth, Quotas, and Expiry

Add agent tokens, per-token quotas, max upload size, default TTL, max TTL, deletion, and audit logs.

Operational Finish

Add health checks, structured logs, metrics, backup policy, admin list/revoke page, and deployment documentation.

Open Decisions

  • Should uploaded HTML allow JavaScript, or should plans be static-only by default?
  • Should URLs be public bearer links, password-protected, or visible only through Cloudflare Access?
  • What default expiry makes sense: 24 hours, 7 days, or manual deletion?
  • Do agents need image/file asset uploads, or should everything be embedded in one HTML document?
  • Is this for personal use only, or multi-user/team use with separate agent identities?
Suggested first implementation: Cloudflare Worker + R2 + D1, static-only HTML previews, seven-day default TTL, 1 MB upload cap, separate preview hostname.
